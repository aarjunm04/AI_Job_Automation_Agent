# AI Job Automation Agent: Phase 2.5 Refactor Tasks

> **Context for Next Session:** 
> The core architectural shift for this refactor is migrating away from flat, disconnected JSON configuration files (`config/*.json`) and sprawling OS environment variables (`java.env`) into a **PostgreSQL Single-Source-of-Truth Architecture**. 
> All dynamic configuration, site/platform limits, user preferences, and profile data must be centralized into two static JSON packets within the `users` table: `user_profiles` (or `preferences_json`) and `site_settings_json` (needs to be added). 
> 
> **Data Isolation Rule:** 
> Static configuration data (`user_profiles`, `site_settings`) must NEVER be mutated during a pipeline run by agents. They have a strict **one-way read path** for the backend systems, and a strict **one-way write path** from the Chrome Extension/User UI.

## 🧠 Session Context Dump (For Future IDE Chats)

**The Core Shift: "Single Source of Truth Architecture"**
Previously, configurations like `run_config` (jobs per run, dry run), search queries, and platform rate limits were scattered across multiple places:
1. `java.env` (loaded into OS env vars and parsed into `config/settings.py`)
2. `config/platforms.json`
3. `config/search_queries.json` 
4. `config/user_preferences.json`
5. `scrapers/job_filters.yaml`

This presented synchronization issues between the automated Docker cron run and the user's manual Chrome Extension actions. 

**The Solution:**
We are eliminating all of the above static config files. Instead, all configuration and preference data will exist completely within the PostgreSQL `users` table as two JSONB columns: 
- `user_settings` (for profile info, run limits, target jobs, resume targets)
- `platform_settings` (for platform-specific rate limits, search strings, and global weights)

These two JSON packets are intended to have a:
- **Strict ONE-WAY Read Path:** The system backend (`MasterAgent`, `FastAPI Server`, `Scrapers`) ONLY reads from these tables and parses them into internal config objects at runtime. Agents NEVER modify these settings dynamically during a run.
- **Strict ONE-WAY Write Path:** Only the user can modify these settings via pushing updates to two new JSON files in the repo (`config/user_settings.json` and `config/platform_settings.json`). A GitHub Action (`config_sync.yml`) will listen for changes to these files upon `git push` and execute an `UPDATE users SET...` query via `psql` to synchronize the live database with the repo state. 

**Component States & Known Bugs Fixed in this Context:**
1. **AgentOps tracing:** In `.py` files using `CrewAI tools`, calling tools via standard Python invocation (e.g. `query_resume_match(x)`) throws `TypeError: 'Tool' object is not callable` in `CrewAI > v0.90` and `AgentOps > v0.4`. The tools must be invoked by explicitly calling `.func(x)` to bypass the CrewAI wrapper. This primarily affects `apply_agent.py` and `analyser_agent.py`.
2. **Chrome Extension:** We discovered the `manifest.json` had "MCP" references and a fragmented UX strategy. We decided on a persistent Sidebar layout connecting *exclusively* to the `FastAPI` instance. It should never talk directly to the DB or Agents.
3. **Docker:** The current `Dockerfile` tries to cram Playwright/Chromium UI dependencies *and* CrewAI LLM Agent dependencies into one mega-image (`agentrunner`). This leads to OOM crashes. The new architecture splits them: one dedicated scraper image, one lightweight `agent-runner` image, and one `api-server` image.

When executing the prompts above, explicitly remember that the JSON files `config/user_settings.json` and `config/platform_settings.json` now act purely as *sync sources* for the GitHub Action, NOT as file-system config files read by the Python application. The Python application *must* fetch the values directly from the Postgres database.
